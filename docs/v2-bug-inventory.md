# V2 State Machine — Known Bug Inventory (pending doctor signoff)

This document lists the inherited quirks in `lib/v2/treatment-state-machine.ts`
that V9 faithfully reproduces by construction (per the V9 voice-clone plan).
Each entry requires doctor signoff before we fix it in V2, because changing
V2 changes what the app says to the patient.

> V9 inherits these bugs on purpose. The fix path is always "edit V2; let
> parity CI confirm V9 followed." Do not patch V9 locally.

Signoff process:

1. Doctor reviews each row below and marks `Fix Priority`.
2. Engineering opens a PR that edits **only** `lib/v2/treatment-state-machine.ts`.
3. The V2-V9 parity gate (`npm run ci:v9-gate`) proves V9 inherits the change
   with zero drift.
4. Route-level parity spec (`npm run test:v2-v9-route-parity`) confirms no
   side-effects under a real HTTP boundary.

## Bugs

### B1 — Triple-nesting return-stack corruption

- **Area**: `determineNextStep` / digging-deeper return tracking.
- **Symptom**: Triple-nested digging flows (e.g. Trauma → Blockage →
  Problem) lose the outermost return point because `returnToDiggingStep`
  is a single scalar and gets overwritten by the middle layer.
- **Repro (engineering)**:
  1. Start Trauma Shifting, define a trauma.
  2. Answer YES at `trauma_dig_deeper_2`, define issue 1, select
     Blockage Shifting.
  3. Complete Blockage, answer YES at `scenario_check_1`, define issue
     3, select Problem Shifting.
  4. Complete Problem Shifting.
  5. Observe: flow returns to `scenario_check_1`, then on next "no"
     the originally expected `trauma_dig_deeper_2` return point is gone.
- **Proposed fix**: Replace the scalar with a stack
  (`context.metadata.diggingReturnStack: string[]`). Full patch lives in
  [`triple_nesting_fix.md`](../triple_nesting_fix.md); implementation
  requires `pushDiggingReturn` / `popDiggingReturn` helpers and a
  migration layer for in-flight sessions.
- **Estimated blast radius**: Medium — 2 files, ~100-150 LOC, touches
  every modality completion handler.

- **Doctor signoff**: _[initials + date]_ ___

### B2 — `clear_anything_else_problem_1` vs `_2`/`_3` inconsistency

- **Area**: `anything_else_check_*` flow in V2.
- **Symptom**: The three "anything else" clearing steps disagree on how
  they hand control back to the UI:
  - `clear_anything_else_problem_1` returns the **user-facing string**
    `"We need to clear this problem. Which method would you like to use?"`
    (lib/v2/treatment-state-machine.ts, ≈ line 5302).
  - `clear_anything_else_problem_2` returns the internal routing token
    `"METHOD_SELECTION_NEEDED"` (lib/v2/treatment-state-machine.ts,
    ≈ line 5352), same as `_3`.
- **User impact**: On the first "anything else" loop the patient sees a
  sentence that they do not see on loops 2 or 3; loops 2/3 instead
  surface method-selection buttons via the routing token. V9 mirrors
  both behaviors exactly.
- **Proposed fix**: Pick one canonical behavior (either always speak
  the sentence _or_ always emit the routing token and rely on the UI)
  and apply to all three. Doctor decides which is the intended flow.
- **Estimated blast radius**: Low — one scripted response per branch,
  3 call sites total, behavior already idempotent.
- **Fix Priority**: _[doctor to mark: P0 / P1 / P2 / won't fix]_ ___
- **Doctor signoff**: _[initials + date]_ ___
